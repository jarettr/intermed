# Contributing to InterMed

Thanks for looking at the code. This file is the map: how the project is built and
checked, how it is laid out, the invariants every change has to keep, and how to
add the common kinds of analysis. It is meant to let you make a change that fits
the grain of the codebase rather than fighting it.

InterMed is `0.1.0-alpha` and moves fast; treat APIs between crates as unstable.

## Principles

The same four principles govern the code and its docs:

- **Honesty** — report only what the evidence shows. No finding without a fact on
  disk to back it; absence of evidence is never proof. When a check cannot be
  sure, it says less, not more.
- **Convenience** — the tool and its output should be navigable; the docs are
  layered (README → guides → reference) so a reader goes only as deep as needed.
- **Concreteness** — concrete commands, flags, ids, examples; not vague prose.
- **Quietness** — understated by default. Document and code-comment limitations in
  equal measure with capabilities. Never oversell.

If you change behavior, the docs in `docs/` change in the same PR.

## Building and checking

```bash
cargo build                 # debug; also writes the man page + completions (build.rs)
cargo test                  # the full suite
cargo build --release       # use the release binary for anything performance-sensitive
```

The release build matters: debug builds are several times slower on real packs,
so benchmark and reproduce scale issues with `--release`.

CI (`.github/workflows/ci.yml`) gates every push and PR on, in order:

1. `cargo fmt -- --check` — formatting is enforced; run `cargo fmt` before pushing.
2. `cargo clippy --all-targets -- -D warnings` — warnings fail the build.
3. `cargo build` + checks that `docs/man/intermed.1` and the four shell
   completions exist (they are generated by `build.rs`).
4. `cargo test`, then the mod-scale benchmarks (`mod_scale_bench`, 50/100/200
   mods).
5. A separate job for the DuckDB feature (`--features duckdb`).
6. `cargo audit` against the RustSec database; accepted unmaintained-only
   advisories live in `.cargo/audit.toml`.

Match all six locally before opening a PR. The man page and completions only
regenerate into `docs/` when you build with `INTERMED_GENERATE_CLI_DOCS=1`; do that
and commit them when you change the CLI surface.

## How the analysis is shaped

Everything is one pipeline with a strict contract:

```text
target → collectors → Facts → rules → Findings (each carrying its Evidence)
```

- **Collectors observe; they never judge.** A collector reads a file and emits
  [`Fact`]s. It does not decide anything is wrong.
- **Rules decide.** A rule reads facts and emits a `Finding`, and every finding
  carries the `EvidenceEdge`s that justify it — down to the jar and the file
  inside it. That chain is what `--explain <id>` prints.
- **Typed-AST layers never emit findings.** The resource AST (Layer M) and the
  mixin model parse meaning into facts; the *rules* turn facts into findings. Keep
  that seam — it is what keeps analysis auditable.
- **Degrade conservatively.** Where a check loses information (a missing mapping,
  an unrecoverable stack frame, a loop in dataflow), lower the confidence and say
  so; never invent a concrete claim you have not proven.

`intermed-doctor-core` owns the pipeline and its contracts; everything else plugs
into it.

## Crate map

The analysis is organized as lettered **layers** (`A`–`K`, plus `M`; there is no
`L` — the runtime-preflight layer was removed and should not be revived). Each is
a crate that emits facts; rules over those facts live in `intermed-rules`.

| Layer | Crate | What it does |
| --- | --- | --- |
| A / B | `intermed-minecraft-scan` | Environment detection (A) and mod/plugin metadata (B) — pure-Rust jar/manifest/bytecode reading. |
| C | `intermed-deps` | Dependency & version reasoning: pairwise checks plus a PubGrub global resolver with unsat derivation trees. |
| D | `intermed-log` | Crash-report / `latest.log` pattern matching. |
| E | `intermed-vfs` | Resource/data collision analysis over a virtual filesystem (byte-level: identical / override / safe union). |
| E (dynamics) | `intermed-dynamics` | What scripts (KubeJS/CraftTweaker) *remove* at datapack load — keyword-and-literal reads, not an interpreter. |
| F | `intermed-mixin-intel` | Static mixin intelligence (see [the engine of it](docs/guides/mixins.md)). |
| G | `intermed-security-audit` | Static security surface — detection only, no enforcement. |
| H | `intermed-sbom` | SBOM / provenance / signature & trust heuristics. |
| I | `intermed-spark-bridge` | Imports a Spark performance profile; never runs the profiler. |
| J | `intermed-rules` + `intermed-columnar` + `intermed-duckdb` | Declarative rule packs and the backends that run them (see [the query engine](docs/reference/engine.md)). |
| K | `intermed-lab` | Compatibility Lab: pinned corpus, smoke-test ingestion, compatibility matrix. |
| M | `intermed-resource-ast` | Typed resource/data semantics over the VFS (recipes, tags, loot, worldgen, …). |

Supporting crates: `intermed-facts` (the ground-truth fact model), `intermed-evidence`
(findings + evidence edges), `intermed-doctor-core` (the pipeline),
`intermed-report` (terminal / JSON / SARIF / HTML rendering — pure, never
recomputes), `intermed-config`, `intermed-resource-identity` (one definition of
"what namespace owns this path"), `intermed-packops` (the only crate that writes,
and only on `vfs overlay`), `intermed-query-bridge` (a test-time guard that the
columnar engine and the shared interpreter emission stay in agreement), and
`intermed-cli`.

## Adding analysis

- **A new finding from existing facts** — write a declarative rule in
  `intermed-rules`. A `FactFinding` rule (scan a kind, keep rows matching a
  conjunction, emit) runs on every backend unchanged. Add a test that asserts the
  finding ids on a fixture.
- **A new fact** — emit it from the owning layer's collector, give it a stable
  attribute shape, and document it in
  [`docs/reference/facts.md`](docs/reference/facts.md). Bump the layer's cache
  version (each layer exposes one) so cached scans re-run.
- **A new resource domain** — add a parser under
  `intermed-resource-ast/src/domain/`. It produces a typed summary and reference
  edges only; it emits no findings. For a small/uniform datapack registry, prefer
  a declarative entry in the registry-spec table over a bespoke parser.
- **A new CLI surface** — extend `intermed-cli`, then regenerate the man page and
  completions (see above) and update
  [`docs/reference/commands.md`](docs/reference/commands.md).

When in doubt about behavior the docs assert, the ground truth is `--help`,
`--dump-config`, and the tests — not older prose.

## Tests, corpus, and fuzzing

- Unit and integration tests live with their crates; the CLI has end-to-end
  tests. New analysis needs a fixture and an assertion on concrete finding ids.
- The `fuzz/` directory holds the libFuzzer targets for the untrusted parsers
  (zip, class files, JSON, mappings). If you touch a parser on the
  crafted-input path, run the matching target.
- Behavior changes to a backend must keep the backends in agreement: the columnar
  engine is proven to select the same facts and emit the same findings as the
  interpreter, and that equivalence is asserted on real packs. Do not let it
  drift.

## Reporting bugs and vulnerabilities

A wrong finding (false positive/negative) is a normal bug — open an issue with the
input and the `--explain` output. A crafted input that crashes or escapes the
sandbox is a security report; see [`SECURITY.md`](SECURITY.md) and report it
privately.
